# FusionMob

**FusionMob** is an **Autodesk Fusion 360 add-in** for parametric furniture (cabinet) design that also exports a cut list ready for import into **CorteCloud** (a Brazilian panel-cutting / nesting service). It targets Brazilian *marcenaria* (custom cabinetmaking) workflows.

## What it does

Inside Fusion 360, it adds a **FusionMob** ribbon tab (in the Solid environment) with a **Cabinet** panel containing four commands:

| Command | Purpose |
|---|---|
| **New Panel** | Create a single parametric panel (board) with dimensions, material, function, and edge-banding (*fita*) per edge. |
| **New Cabinet** | Generate a full frameless cabinet carcass: two sides, base, top, N evenly-spaced shelves, an optional grooved back panel, an optional toe kick (*rodapé*), optional doors (*portas*), and optional drawers (*gavetas*) with embedded slide hardware. |
| **Edit Cabinet** | Pick an existing cabinet, tweak any parameter, and regenerate it in place. |
| **Export Cut List** | Collect every tagged panel in the document and write a CorteCloud-compatible CSV. |

## Architecture

Two conceptual layers:

- **Layer 1 (geometry):** Builds 3D solid bodies (one component per panel) and stores each panel's cut-list definition as JSON in a Fusion **body attribute** (group `FusionMob`, name `panelData`). Cabinets also store their full creation config as JSON on the cabinet component (`cabinetConfig`), enabling non-destructive edit/rebuild.
- **Layer 2 (export):** Reads those stored JSON attributes back out (`findAttributes`) and writes the CorteCloud CSV.

Geometry is created with `TemporaryBRepManager` boxes (with boolean-difference cuts for dado grooves and toe-kick notches) so parts land at exact coordinates. Panels are locked together via as-built rigid joints, falling back to a rigid group. Drawer **slide hardware** is the one exception to the build-from-scratch rule: it is either a lightweight parametric proxy box or a real CAD model imported from the bundled hardware library (see *Drawers* below).

### Key domain details
- **Units:** UI/CSV use millimeters; Fusion's internal geometry uses centimeters (the code converts by ÷/×10).
- **Dimensions are FINISHED sizes** — CorteCloud deducts edge-tape thickness itself, so the add-in does not pre-deduct. Thickness is encoded in the material name (e.g. `MDF 18mm Branco`).
- **Back panel** seats in dado grooves cut into the sides/base/top, with configurable clearances.
- **Toe kick** is a separate box the carcass rests on (standard BR practice), with the cabinet's overall height *including* the base.
- **Doors** (*portas*) are frameless: N equal doors on the front face, separated from each other by an even reveal gap (*folga*), edge-banded on all four edges and grain-locked (`Girar=Nao`). A `door_inset` flag (checkbox *Porta embutida (inset)*) picks the mounting style: **overlay** (*sobreposta*, default) mounts the doors forward of the Y=0 front plane, spanning the full carcass width inset from the edges by the gap; **inset** (*embutida*) seats them inside the clear opening (width `W−2t`, height `Hbox−2t`), flush with the front face, with the gap as the reveal all round. Either way they cover the carcass box front (not the toe kick). **Shelf depth** has two modes, chosen by the `shelf_align_front` flag (checkbox *Prateleiras alinhadas com a frente*): **shallower** (default) recesses every shelf from the front by `shelf_front_setback` (default 30 mm), while **aligned** (flag on) makes them flush with the carcass front (base setback 0). On top of the chosen base setback, shelves are always pushed back far enough that a closed door clears them (an inset door reaches `door_t` into the opening, an overlay door needs only a small gap — the `shelf_door_clearance` tolerance); the effective setback is `max(base_setback, door reach)`, so even frame-aligned shelves never conflict with an inset door. Doors are **not** part of the rigid carcass — each gets its own vertical **pivot (revolute) joint** on its outer vertical edge (hinge) so it opens like a real hinged door. Doors left of centre hinge left, the rest hinge right (a two-door pair meets in the middle). Since both hinge edges run +Z, a positive rotation swings both the same rotational sense; the swing is therefore limited per side — right-hinged doors `0…+110°`, left-hinged doors `−110…0°` — so each only opens outward.
- **Hinge boring** (concealed *dobradiça de caneco*, optional `with_hinges`, on by default when doors are enabled) is **model-only geometry**, cut with the same `TemporaryBRepManager` boolean-difference path as the dado grooves — cylinders instead of boxes (`_make_temp_cylinder`; `add_solid_panel` now takes a `holes=` list of `(x0,y0,z0,x1,y1,z1,radius)` cm tuples). Each door gets a blind **35 mm cup bore** in its interior-facing back face (Y=0 for overlay, Y=+`door_t` for inset), inset from the hinge-side edge; the hinge **count scales with door height** (`hinge_count_for_height`: 2 ≤900 mm, 3 ≤1600 mm, 4 ≤2000 mm, else 5), first/last inset 100 mm from the ends and the rest evenly spaced. The matching **mounting-plate screw pilots** are bored into the adjacent side panel's interior face — but only for the **outermost doors**, whose hinge edge sits at a real side panel; a frameless carcass has no interior divider for inner doors (>2) to mount to, so those get cups only. Boring dimensions are **per-cabinet** (`cfg['hinge']`, defaults from the module-level `HINGE` dict, filled in by `normalize_cfg` for older stored configs); five knobs — cup diameter, cup depth, edge-to-cup-centre, end inset, and shelf clearance — are editable in the **Advanced** section, while the plate screw-pilot specifics stay at the `HINGE` defaults. **None of this reaches the CSV** — CorteCloud's format has no *furação* field — so the per-door hinge count is appended to the door's `Complemento` as a note instead.
- **Hinge/shelf collisions:** the only real clash is a hinge plate landing at the same height as a shelf (both mount to the side panel's interior face; the cup bore itself never collides, as it cuts into the door toward −Y while shelves sit at +Y). Shelf positions (`shelf_z_bottoms`, shared by the builder and the check so they can't drift) are compared against the hinge Z's; any hinge within `HINGE['shelf_clearance']` of a shelf band is **auto-nudged** to the nearest clear position within the door (`resolve_hinge_conflicts`) — the cup and plate share the same nudged Z so they stay aligned. A hinge that can't be cleared (a crowded door) is left in place and reported via a non-fatal note in the New/Edit Cabinet result dialog (`build_cabinet` returns a `warnings` list).
- **Drawers** (*gavetas*) are a single column of N stacked drawers across the carcass front, laid out like doors via a `drawer_inset` flag (overlay default / *embutida*). Each drawer is a full BR-standard **box** — 2 sides + a box front + a box back (from `drawer.box_t`, default **16 mm** — the side thickness the UM A30 is designed for; functions `Lateral`/`Travessa`) plus a **bottom** seated in a dado groove cut into all four walls (`drawer.bottom_t` ≈6 mm, `Fundo`), same boolean-groove path as the back panel — plus a separate **face** (*frente*, `drawer.face_t` ≈18 mm, function `Porta`, banded all four edges, `Girar=Nao`). All six are tagged `panelData` → they go in the cut list. Box sizing is driven by the slide's real planning spec: the drawer bottom **between the sides = clear opening − `carcass_deduction`** (42 mm for the Matrix Invisa A30 — the fixed amount undermount runners need), so the outer box width = `inner_w − carcass_deduction + 2·box_t` and the box is centred with an equal air gap per side (≈5 mm with the default 16 mm sides the UM A30 is designed for); box depth = `min(recommended_box_depth (≈NL), back-of-cabinet − back_clearance)`; and `validate_cfg` enforces the slide's `min_cabinet_depth` (318 mm for the 300 mm NL). **Faces** stack over the front region (overlay = full carcass front, inset = clear opening) with the reveal `drawer_gap`; each **box** is laid out behind its own face: the box floor sits a `bottom_clearance` runner gap above the face bottom (but never below the carcass base), and the box top stops `drawer.box_top_gap` (~30 mm) below the face top / interior ceiling, so a short box rides behind the tall face and never fouls the drawer above. Aligning the box to its face (rather than to an independent interior grid) is what keeps the bottom drawer from sitting too high. Box side height is whatever that leaves, capped by `drawer.box_height` (default 150 mm) for very tall single drawers. Box/bottom sub-specs live in the module-level `DRAWER` dict (deep-merged in `normalize_cfg` like `hinge`/`tol`); only the two materials + face band are exposed in the dialog. **The model is organised into components under the cabinet**: the carcass box (`Corpo`) and the toe kick (`Rodape`) are sub-assemblies of their panel-components, each locked with a rigid group (`_rigid_group_occs`); each drawer is **one component (`Gaveta N`) whose parts are bodies** (via `add_solid_body`, not child components) so the whole drawer is inherently rigid and can take a slider joint off its own geometry. Everything is still built at the same cabinet-local coordinates (all these components carry identity transforms). The carcass + toe kick are rigid-grouped into one grounded structure; doors pivot and each drawer gets a **slider (prismatic) joint** — anchored to the grounded `Corpo`, its axis a Y-running edge of the drawer's own box-side body (`attach_drawer_slides` + `_find_horizontal_y_edge`; as-built joints need real geometry for non-rigid motion, so `None` — which only works for rigid joints — silently fails). Travel is limited to `[0, 0.9·box_depth]` (positive along that edge moves the drawer forward) so it opens toward the front and stops near full extension. **Drawers and shelves are mutually exclusive** in one opening (validation rejects the combo), as are overlay-doors + overlay-drawers.
- **Drawer slide hardware** is chosen from a bundled **library manifest** (`resources/hardware/hardware.json`, keyed by `slide_key`, seeded with a *Häfele Matrix Invisa A30 GT2 300 mm* undermount full-extension push slide). `resolve_slide_spec(cfg)` returns the full spec (all mm) backfilled from the module-level `FALLBACK_SLIDE`, so the mounting spec (`carcass_deduction`, `bottom_clearance`, `back_clearance`, `min_cabinet_depth`, `recommended_box_depth`; `side_clearance`/`side_panel_thickness`/`base_depth_offset` are informational) lives in the manifest — **not** the cabinet config — and switching slides re-sizes the boxes automatically (the hardware analogue of how `HINGE` defaults fill in). Representation is **hybrid**: ticking *Inserir modelo 3D da corredica* (`insert_real_hardware`) imports the real bundled CAD; otherwise a lightweight parametric **proxy** box (`make_slide_proxy`) marks each slide — sitting in the runner gap directly under the box (height = `bottom_clearance`, length `proxy_L` along Y, width `proxy_W`), hugging each box side. The real model is imported **once, up front, before the cabinet component exists** (`import_hardware_component`) — CAD (`.step/.stp/.f3d/.iges`) via `ImportManager.importToTarget` into the **root** (importing into a freshly created child mid-build invalidates the in-progress assembly — Fusion then raises *"refers to a deleted Object"*), mesh (`.obj/.stl/.3mf`) via `meshBodies` in a child holder. That imported component is then **instanced** (`occurrences.addExistingComponent`, positioned by `_matrix_from_transform` from the spec's `model_transform`) at each slide slot at the end of the build, grounded, and the temporary root import is deleted; multi-drawer docs stay light because the geometry is shared. Import is guarded — a missing/failed model just falls back to the proxy, never aborting the build. **Neither proxy nor model carries `panelData`**, so hardware never reaches the CSV (like hinges); the chosen slide is appended to the drawer face's `Complemento` instead. Bundled models go under `resources/hardware/models/` and should be **closed-state STEP** (clean, light); the imported real model is fixed (grounded), since a single body's rails aren't separable, while the drawer box slides.

## CSV format (CorteCloud "importar do Excel")

Semicolon-delimited (`;`), UTF-8 with BOM. Columns:

```
Quantidade;Comprimento;Largura;Funcao;Fita C1;Fita C2;Fita L1;Fita L2;Material;Complemento;Girar
```

`Comprimento` is the larger face dimension, `Largura` the smaller. `Girar` (`Sim`/`Nao`) tells CorteCloud whether it may rotate the part during nesting (grain lock). See [cortecloud_importar.csv](cortecloud_importar.csv) for a sample export.

## Layout

```
FusionMob/
├── FusionMob.py         # entire add-in (run/stop lifecycle, commands, geometry, export)
├── FusionMob.manifest   # Fusion 360 add-in manifest (type: addin, Windows|Mac)
├── resources/           # 16x16 / 32x32 command icons (new_panel, new_cabinet, edit_cabinet, export)
│   └── hardware/         # drawer-slide library: hardware.json manifest + models/ (bundled CAD)
└── .vscode/launch.json  # Fusion 360 Python debugging config
cortecloud_importar.csv  # sample exported cut list
```

## Versioning

The add-in version lives in **two places that must stay in sync**: `version` in [FusionMob.manifest](FusionMob/FusionMob.manifest) and `__version__` in [FusionMob.py](FusionMob/FusionMob.py). It follows semantic versioning (`MAJOR.MINOR.PATCH`).

**IMPORTANT: increment the PATCH version (the last digit) on every modification to the add-in.** Bump MINOR for new features and MAJOR for breaking changes, resetting the lower digits as usual. Always update both files together.

## Development notes

- Single-file Python add-in using the Fusion 360 API (`adsk.core`, `adsk.fusion`). Entry points: `run(context)` (registers UI) and `stop(context)` (tears it down). The one runtime data dependency is `resources/hardware/hardware.json` (the drawer-slide library), loaded lazily and cached; it degrades to the built-in `FALLBACK_SLIDE` if absent, so the add-in still runs without it.
- Requires an **Assembly or Hybrid** design (not a Part design — those allow only one component); the code raises a friendly `PartDesignNotSupportedError` otherwise.
- Comments and UI labels mix English and Portuguese, matching the target audience.
- Began as "Phase 1" (parametric panel editor + cut-list exporter); has since grown a full carcass generator, doors with concealed-hinge boring, and drawers with embedded slide hardware (the first feature to import external CAD rather than build geometry from scratch).
- **Importing external CAD** (real drawer-slide models) is the exception to the from-scratch geometry rule — prefer lightweight closed-state STEP; mesh imports (`.obj/.stl/.3mf`) work but are heavy, and multiple drawers × 2 rails can slow rebuilds, which is why the proxy is the default and `insert_real_hardware` is opt-in.
